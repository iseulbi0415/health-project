# health-project — 프로젝트 개요 (Claude Code용 컨텍스트)

## 프로젝트 목표
명지대학교 '2026년도 제5회 창의적 SW프로그램 경진대회' SW프로그램 개발 부문 출품작.
역류성 식도염 케어 + 러닝 습관 관리를 위한 개인 건강관리 웹앱.

## 대회 일정
- 참가신청 마감: 7/22
- 완성작품 제출 마감: 8/14
- 1차 서류심사: 8/18~24
- 2차 발표심사: 8/28 (참가자 직접 시연 방식)

- AWS 배포는 제출 요건 아님 (로컬 실행으로 충분)

## 심사 기준 관련 (중요)
선배 조언: 심사는 창의성 > 완성도 중심이며, 코드 한 줄 한 줄보다 전체 아키텍처/설계 이해가 중요.
개발자(본인 또는 AI)가 코드를 짜더라도, 아래는 반드시 본인이 완전히 이해하고 설명할 수 있어야 함:
- 전체 시스템 아키텍처 (프론트-백엔드-DB 구조)
- 사용한 기술적 방법론과 그 이유
- API 명세 (엔드포인트, 요청/응답 형식)
- 사용자 시나리오 (화면 간 흐름)

## 핵심 기능
1. 소화 안전 타이머 — 식사 완료 시 안전하게 누울 수 있는 시간 카운트다운
2. 식단 & 칼로리 기록 — 자주 먹는 음식 빠른 추가 + 총 섭취 칼로리 자동 합산
3. 트리거 음식 경고 — 역류 유발 음식(고지방/튀김, 매운음식, 산성음식, 카페인, 초콜릿, 자극성 향신료, 탄산, 알코올) 표시 (출처: Harvard Health Publishing)
4. 러닝 기록 일지 — 거리, 시간, 페이스, MET 기반 칼로리 계산
5. BMR/TDEE 계산기 (Mifflin-St Jeor Equation 기준)
6. 컨디션 메모장 — 오늘의 컨디션/증상 기록

## 기술 스택
- **프론트엔드**: HTML/CSS/JS (모노레포 루트)
- **백엔드**: Spring Boot (Maven, Java 17, Spring Web + JPA + MySQL Driver + Lombok), `backend/` 서브폴더
- **DB**: MySQL (`health_project`), Homebrew로 설치
- **버전관리**: GitHub (`iseulbi0415/health-project`)
- **에디터**: VS Code(프론트) + IntelliJ(백엔드)

## 작업 워크플로우 (반드시 이 순서로 진행)
1. **파악** — 작업 시작 전 관련 파일/코드를 먼저 읽고 현재 상태 파악
2. **계획 제시 (Plan 모드)** — 새 기능/큰 변경은 파일을 바로 고치지 말고, 어떤 파일을 왜 어떻게 바꿀지 계획부터 보여주고 승인 받은 뒤 진행
   (`Shift+Tab`으로 Plan 모드 전환 — 이 프로젝트에서는 새 기능 구현 시 기본으로 이 모드에서 시작할 것)
3. **구현** — 승인된 계획대로 코드 작성/수정
4. **자기 검증** — 에러가 있으면 원인을 분석하고 스스로 고친 뒤 결과를 보고 (막히면 추측으로 우회하지 말고 원인을 설명)
5. **결과 보고** — 뭘 바꿨는지, 왜 그렇게 했는지 요약. 지언이 이해했는지 확인 후 다음 단계로

## 권한 모드 사용 원칙
- 처음에는 파일 수정 전 매번 확인받는 기본 모드(Ask before edit)로 진행
- 익숙해지고 신뢰가 쌓인 반복 작업(예: 이미 검증된 패턴을 다른 화면에도 똑같이 적용하는 것)에 한해 자동 승인 모드로 전환 가능
- **Bypass permissions 모드는 지양** — 검증 안 된 상태에서 전체 권한을 주는 건 위험

## 프롬프트 작성 원칙 (지언이 Claude Code에게 일 시킬 때)
막연하게 "알아서 해줘"라고 하지 말고, 아래를 구체적으로 포함해서 요청:
- **목표**: 최종적으로 뭐가 되어야 하는지
- **현재 상태**: 지금 뭐가 어떻게 되어 있는지 (관련 파일/함수명 포함)
- **제약/맥락**: 지켜야 할 규칙 (위 코딩 컨벤션, 대회 일정 등)

예: "트리거 음식 태그가 화면에 안 뜨는데, Food.java엔 isTrigger 필드 추가했고
app.js의 renderFoodList()에서 태그 표시 코드도 넣었어. 원인 찾아서 고쳐줘"
→ 이렇게 현재 상태 + 이미 한 작업 + 문제를 구체적으로 말하는 방식 유지할 것
